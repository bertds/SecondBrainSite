Software design tips :

Classes/methods/interfaces

  • Classes should be deep
  • it is not the amount of code lines that's the limit per method/interface
  • don't have shallow classes/methods, just adds complexity
  • don't use skinny layers
  • example of deep interfaces


    • part of the interface is also the exceptions and the amount of times you can call this interface (batch vs online)

    • interface should be simple. you don't have to know to underlying code to know how to call the method.

    • code in method should be one same level/layer of abstraction. so don't do create_customer (high level) and after that change attribute of customer in db (low level).

    • Use information hiding to get deeper modules. This way there is no information leaked about the internal processes of a module or the modules it calls to the outside world. Ensure hower that only unimportant details are omitted from the interface.



    • Define errors out of existence


  • don't throw exceptions whenever possible
  • minimize the amount of places you habe to handle exceptions
  • think about the caller and redefine method semantics to eliminate exceptions (for exampel : delete var if it doesn't exist don't throw exception when not found but redefine as method to make var not exist )

Tactical vs Strategic Programming


Tactical - goal = make it work > system turns spaghetti over time > hard to fix because it is not one problem it is many thousand comulated over time

Strategic - goal = great design > pays back in the future, drive complexity out of system
Design it twice to see if there is a better design possible.

Documentation

  • Put documentation where you will find it > in the code (KBC : docstring in sphynx)
  • Write comments of method first before writing code.
  • Use different words to write the comments than in the code. So don't just repeat what is in the code !
  • Focus on what and why of the code and not how (that's the code)

Performance

First make it work then look at performance.
Meaasure to proof that the performance is better.

How much to invest ?

  • most startups are totally tactical
  • rewriting almost never happens
  • good design culture > hire better developers
  • 10-20% overhead : as investment
  • small steps : write documentation as you're going, write uniitests,
  • when making changes you won't get it right from the first time. always look for something to make better.
  • try to find clean ways of doing things

Is the course working ?

  • make classes a bit more general purpose even if you only use them once to make them simpler and deeper.

  • Provide good defaults in interface/method so users don't have to fill in the most common used combination of parameters.
